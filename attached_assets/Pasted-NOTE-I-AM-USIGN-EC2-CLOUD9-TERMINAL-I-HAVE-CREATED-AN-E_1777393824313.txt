I found a dedicated shared-feed architecture (FeedServer/FeedClient) plus dual global providers. I’ll now verify broker-specific static-IP and whether any client broker still opens its own market data feed unexpectedly (which could cause your runtime issues).


How global data feeder is wired (actual flow)
Global market data is created in ProviderFactory from data_providers (Upstox + Dhan), not from each client broker.
FeedServer (in web/server.py startup) opens one shared feed stack and broadcasts normalized ticks over local TCP.
Each client bot subprocess uses FeedClient (from ProviderFactory) to receive the same global ticks.
FeedClient republishes ticks as BROKER_TICK_RECEIVED locally; PriceFeedHandler consumes this and updates state for all user sessions (user_id='GLOBAL' fan-out).
Trading logic (TickProcessor -> SellManagerV3) runs from this shared state.
So yes: Dhan/Upstox are the global data feeders for trading engines in the intended design.

How client brokers connect for execution
Client subprocess is spawned by InstanceManager with env (CLIENT_ID, CLIENT_BROKER, credentials, etc.).
BrokerManager.load_client_mode_broker creates broker clients (zerodha, angelone, upstox, dhan, etc.) for order execution.
Sell V3 places orders through broker instances (_place_order_on_broker), while strategy signals use shared feed data.
This separation (shared data + per-client execution) is implemented.
Static IP behavior (important)
Static IP is stored at user/client level (users.static_ip) and propagated to all that client’s broker instances.
So: one client with multiple brokers sharing same static IP is supported (good).
Broker order calls validate IP via BaseBroker._validate_source_ip() and handle AWS EIP/NAT logic.
For SDK sessions that support it, adapter-based source IP binding is mounted; others use scoped patching.
Critical mismatch I found
In admin_api (PATCH /clients/{client_id}/static-ip), system blocks assigning same static_ip to another client (409 conflict).
That means: multiple clients cannot currently share one whitelisted IP in this app, even if broker policy technically allows it.
If your infra expects one EIP shared across many clients/brokers, this API check is currently preventing that model.

Possible runtime issue points (likely what you saw)
If FeedServer is not up/reachable, each subprocess retries and may eventually activate local fallback feed manager (can create extra WS load).
If only one global provider is configured, behavior is less robust than dual provider mode.
OAuth/token update paths can restart per-broker feeds in some flows (BrokerManager.handle_token_update), which may conflict with your “global feeder only” intent if triggered.
Direct answer to your IP question
One client, multiple brokers, same static IP: supported by current code.
Multiple clients, multiple brokers, same static IP: currently blocked by app policy (admin API uniqueness check), not by broker SDK logic itself.
Technically, same public IP can usually be whitelisted for multiple API apps/accounts unless broker contract explicitly mandates dedicated IP per app/account.

# Business-Critical Options-Selling Flow (End-to-End)

This document traces the live, business-critical Sell V3 flow from process startup to order placement, persistence, and square-off.

## Scope and primary path

- Runtime entry: `bot/main.py`
- Engine/session orchestration: `bot/hub/engine_manager.py`
- Per-instrument runtime: `bot/hub/live_orchestrator.py`, `bot/hub/base_orchestrator.py`
- Tick ingestion + dispatch: `bot/hub/price_feed_handler.py`, `bot/hub/tick_processor.py`
- Sell strategy logic: `bot/hub/sell_manager_v3.py`, `bot/hub/sell_v3/entry_logic.py`, `bot/hub/sell_v3/exit_logic.py`
- Broker routing/execution: `bot/hub/broker_manager.py`
- State and PnL storage: `bot/hub/state_manager.py`

## 1) Boot and system initialization

1. `main()` initializes configuration, logging, validation, and graceful-shutdown handlers.
2. It creates core managers:
   - `BrokerManager` (broker discovery and routing)
   - optional Redis and DB managers
   - `EngineManager` (instrument/session lifecycle)
3. In live mode, event bus routes execution requests:
   - `EXECUTE_TRADE_REQUEST` -> `BrokerManager.handle_execute_trade_request`
   - `EXIT_TRADE_REQUEST` -> `BrokerManager.handle_exit_request`
4. `EngineManager.run_engines()` becomes the long-running control loop.

## 2) Pre-market runtime bring-up

`EngineManager.run_engines()` performs staged bring-up before active trading:

1. Loads broker instances (`BrokerManager.load_brokers`) and derives active instruments.
2. Creates market data provider (`ProviderFactory.create_data_provider`) and shared websocket manager.
3. For each instrument:
   - `create_orchestrator(...)` builds `LiveOrchestrator`.
   - `DataManager.load_contracts()` loads options chain/contracts.
   - ATM manager contract lookup tables and expiries are prepared.
   - `finalize_initialization()` creates `TickProcessor` and marks ATM manager ready.
4. Adds user sessions (`BaseOrchestrator.add_user_session`) with isolated per-user `StateManager` and `SellManagerV3` (if V3 enabled).
5. Subscribes websocket to initial instruments (index + futures keys), then idles until session start gate opens.

## 3) Tick ingestion and normalization

Incoming market data reaches strategy logic through `PriceFeedHandler`:

1. Feed handler receives broker ticks and dispatches by instrument key through `tick_dispatcher`.
2. For each dispatched tick:
   - Updates shared and/or user-specific state (`option_prices`, `index_price`, `spot_price`, Greeks, ATP history).
   - Updates aggregators used by indicators (`entry_aggregator`, `exit_aggregator`, 1m/5m aggregators).
   - Updates per-position PnL in `StateManager`.
3. A dedicated worker (`_tick_worker`) coalesces bursts and runs `orchestrator.process_tick()` with timeout protection.

Business meaning: market data is converted into a consistent state snapshot before any entry/exit logic runs.

## 4) Core decision loop per tick

`TickProcessor.process_tick_v2()` is the central gatekeeper.

It performs, in order:

1. Runtime trading toggle checks (client mode):
   - reads DB-backed `trading_active`, mode, quantity.
   - if toggle OFF, forces square-off and halts new entries.
2. Health heartbeat and feed-lag diagnostics.
3. ATM calculation from futures/index and strike watchlist build.
4. User-session buy-side crossover checks and active-trade management (legacy/V2 side).
5. Sell-side invocation:
   - if orchestrator has `SellManagerV3` and trading is permitted, `await sm.on_tick(...)`.

This means Sell V3 receives fully processed tick context after hard operational/risk toggles.

## 5) Sell V3 tick pipeline

`SellManagerV3.on_tick()` runs a strict sequence:

1. Hard stop if `strangle_closed` is already set.
2. Enforces EOD close-time stop; can force full exit and stop-for-day.
3. Updates session indicators/dashboard snapshot.
4. Exit-first policy:
   - if trades exist, calls `ExitLogic.check_exits(...)`.
5. Entry-next policy:
   - only when no active trade and no same-tick exit, calls `EntryLogic.check_entry(...)`.
6. Emits periodic heartbeat in live mode.

## 6) Entry path (opening short CE+PE)

`EntryLogic.check_entry(...)` enforces trading discipline:

1. Pulse/boundary gating (aligned to configured rule timeframes, with live buffer).
2. One-entry-per-candle and per-day trade-limit checks.
3. Session timing gates (priming wait, entry end-time, cooldown after exit).
4. Strategy mode selection:
   - Beginning concept (ATM-anchor based)
   - Re-entry concept (matrix/pool scan)
5. Technical gate evaluation (configured dynamic rules).
6. On pass -> `SellManagerV3._execute_straddle_entry(...)`.

`_execute_straddle_entry(...)` then:

1. Resolves CE/PE contracts from ATM/expiry lookup.
2. Places SELL orders on all eligible brokers (`_place_order_on_broker`).
3. Validates live order success (aborts if all broker orders fail).
4. Persists active-trade state in memory + state file.
5. Increments counters, updates workflow phase, and optionally sends notifications.

## 7) Exit path (risk controls and square-off)

`ExitLogic.check_exits(...)` has layered protection:

1. Highest-priority time exits:
   - `square_off_time` forced EOD close.
2. Global guardrails:
   - cumulative PnL target/stoploss
   - ROC guardrail on timeframe boundary
3. Trade-level controls:
   - single-trade target/stoploss
   - ratio breach
   - scalable trailing lock
4. Smart-roll logic:
   - if enabled, validates re-entry gates before rolling; otherwise full exit.
5. Dynamic technical exits:
   - rule-based reversal exits at configured boundaries.

Actual close is done by `SellManagerV3._execute_full_exit(...)`:

1. Sends BUY orders for open legs across configured brokers.
2. Computes per-leg and total PnL.
3. Appends live trade logs and PnL history.
4. Persists trade history to DB in client mode.
5. Updates session totals and state files; may set `strangle_closed` on stop-for-day exits.

## 8) Multi-tenant isolation and execution routing

- Each user session has isolated state (`UserSession -> StateManager`).
- Sell V3 can run per-user session in client/multi-user scenarios.
- Broker execution is routed to matching user brokers by `BrokerManager` with instrument checks.
- Shared market data is synchronized where required, while trade state remains user-scoped.

## 9) Shutdown and fail-safe behavior

- OS signals trigger graceful shutdown from `main.py`.
- Bot disables orchestrators, calls broker-wide `close_all_positions()`, then tears down tasks.
- Engine loop also closes remaining sell positions when session window ends.
- Event-loop stall monitor warns on severe loop delays.

## 10) Why this is business-critical

This path is the revenue and risk core:

1. It controls when entries are permitted.
2. It decides what strikes are sold and when.
3. It executes orders on real brokers.
4. It enforces mandatory loss/profit guardrails.
5. It persists state/history needed for recovery, audit, and client reporting.

Any regression in this chain can directly impact fill quality, PnL, or risk exposure.
